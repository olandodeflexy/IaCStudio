IaC Studio Documentation

IaC Studio is a local-first workspace for designing, importing, editing, validating, and running infrastructure projects across Terraform, OpenTofu, Ansible, Pulumi, CDK, and CloudFormation export paths.

What IaC Studio Does

Design visually

Drag resources onto a canvas, connect dependencies, edit properties, and preview generated code as the topology changes.

Import real projects

Browse local folders, inspect Terraform, OpenTofu, Ansible, Pulumi, and layered projects, then rebuild the visual graph from disk.

Use AI without giving up control

Run with local Ollama models or API-backed providers. AI can create topologies, explain failures, suggest next resources, and fix plan errors.

Validate before apply

Policy checks, security scans, drift detection, plan gates, command timeouts, and kill controls keep risky operations explicit.

Round-trip code

Canvas edits write IaC files, file changes resync the graph, and the Monaco editor supports Terraform, OpenTofu, Ansible, Pulumi, YAML, and JSON.

Package and promote modules

Inspect module usage, search registry metadata, and promote selected resources into reusable Terraform module structure.

Quick Start

The fastest path is the interactive setup script. It checks dependencies, asks how you want AI configured, builds the app from source, and starts the local server. 

./scripts/setup.sh
What the setup script does: ./scripts/setup.sh does not download the release binaries. It checks your platform, installs missing dependencies when you approve, configures optional AI, builds bin/iac-studio locally, and can start that freshly built local server for you.
What the install script does: scripts/install.sh downloads the latest matching release binary from GitHub, verifies it with checksums.txt when available, installs it as iac-studio, and leaves source builds out of the path.

Manual local install

Use this path when you want predictable commands for a development machine.

git clone https://github.com/olandodeflexy/IaCStudio.git
cd IaCStudio
make deps
make build
./bin/iac-studio

Open http://localhost:3000.

Download a release binary

Release assets are local server binaries. They are useful when you want to run IaC Studio without Docker and without building from source. Download the file that matches your operating system and CPU, make it executable on macOS or Linux, run it, then open http://localhost:3000.

# Optional: rename the downloaded macOS or Linux asset first
mv iac-studio-darwin-arm64 iac-studio

./iac-studio --projects-dir "$HOME/iac-projects"

On macOS, unsigned release binaries may be quarantined by Gatekeeper after download. If the binary is blocked, remove the quarantine attribute and run it again. 

xattr -d com.apple.quarantine ./iac-studio
./iac-studio

checksums.txt is for verifying download integrity. It is not an executable and does not start the app.

Install the latest release with a script

Use this when you want the release binary path but do not want to manually pick the asset. The installer detects your OS and CPU, downloads the matching asset, verifies the checksum, and installs iac-studio. 

curl -fsSL https://raw.githubusercontent.com/olandodeflexy/IaCStudio/main/scripts/install.sh | bash
iac-studio

Run with Docker

The container includes the server, built web UI, Terraform, and OpenTofu.

docker run --rm -it \
  -p 3000:3000 \
  -v "$HOME/iac-projects:/projects" \
  ghcr.io/olandodeflexy/iacstudio:latest
Default server behavior: the binary binds to 127.0.0.1:3000, stores projects in ~/iac-projects, and expects Ollama at http://localhost:11434 unless you override flags or AI settings.

First 10 Minutes

  1. Start IaC Studio. Run ./bin/iac-studio or the Docker command, then open the browser.
  2. Create or import a project. Start from an empty project, describe a topology, upload a diagram, or browse to an existing folder.
  3. Pick the tool and provider. Choose Terraform, OpenTofu, Ansible, Pulumi, or a layered multi-environment layout.
  4. Add resources. Use the left resource palette, AI chat, or topology builder. Connect nodes to encode dependencies.
  5. Review generated code. Open the Code tab, inspect files, and save edits back to disk.
  6. Validate. Run policy checks and security scans before running infrastructure commands.
  7. Run safely. Use Init, Plan, then Apply. Apply requires a successful plan and explicit confirmation.

Core Workflows

Create infrastructure from the canvas

Choose a provider, search the resource catalog, add nodes, then connect resources. Connections are not decorative: for Terraform and OpenTofu they become real references where the generator can infer safe relationships.

Import an existing project

Use the Import Wizard to browse local folders. IaC Studio scans supported files, detects the tool, parses resources, restores known coordinates, and writes a .iac-studio.json state file next to the project.

Build from a text topology

Describe the target architecture in natural language. The AI topology endpoint returns resources and edges, then the UI turns them into editable canvas objects and generated code.

Build from an image

Drop one or more architecture diagrams into the Vision import flow. The backend sends the image payloads to the configured AI provider, converts the result into a topology, and keeps the generated project editable.

Round-trip Pulumi and layered projects

Pulumi TypeScript resources are parsed from index.ts files, while layered projects use environment-aware folders such as environments/dev and shared module paths. The environment selector limits the canvas and code view to the active layer.

Run policy and security checks

Run policy engines and scanners before executing infrastructure commands. Blocking findings stop apply unless the API receives an explicit acknowledgement for the current request.

Detect drift and export

Open the Drift tab in the right panel to compare Terraform/OpenTofu state against code when state is present. IaC Studio classifies each finding and recommends whether to investigate, codify, import, or revert. Known provider noise can be muted with explicit drift.suppressions rules in .iac-studio.json; suppressed findings remain visible in the Drift tab but no longer count as active findings. Export converts the current topology into alternate formats such as Pulumi TypeScript, CDK Python, and CloudFormation where supported. 

{
  "drift": {
    "suppressions": [
      {
        "address": "aws_s3_bucket.logs",
        "path": "tags",
        "classification": "legitimate_config_change",
        "reason": "provider-managed owner tag"
      }
    ]
  }
}

Cloud Connections

Cloud Connections are named run targets for real infrastructure work. They let you keep provider authentication explicit, test the connection before use, and attach a selected account, subscription, project, and region to init, plan, apply, import, drift, and future agent workflows.

Recommended model: use cloud-native local auth first. Prefer AWS profiles or SSO, Azure CLI login, and gcloud auth. Use static secrets only when there is no better option for the environment you are testing.

How to use a connection

  1. Create a named target. Open the Cloud tab in the right inspector, choose AWS, Azure, or GCP, then save the metadata needed for that auth method.
  2. Test it before running commands. The test action verifies required local fields and reports missing values in plain language. Today this is readiness validation, not a full IAM permission simulator.
  3. Select it for runs. Click Use for runs. The app header shows the selected cloud target so you can see the account context before using plan or apply.
  4. Run normal IaC commands. IaC Studio passes provider-specific environment variables only to that command process. The terminal and WebSocket payloads show the public connection name, not secret values.
Provider Preferred paths Fallback path Command environment
AWS aws_profile, aws_sso aws_static AWS_PROFILE, AWS_SDK_LOAD_CONFIG, or static key env plus region.
Azure azure_cli azure_service_principal ARM_SUBSCRIPTION_ID, ARM_TENANT_ID, service principal env when provided.
GCP gcp_gcloud gcp_service_account GOOGLE_PROJECT, GOOGLE_CLOUD_PROJECT, GOOGLE_CREDENTIALS, region env.

Provider setup examples

AWS profile or SSO is the cleanest local path because IaC Studio only needs the profile name.

aws configure sso
aws sso login --profile platform-dev

Azure CLI login keeps refresh and token storage in Azure tooling.

az login
az account set --subscription 00000000-0000-0000-0000-000000000000

GCP gcloud auth lets Application Default Credentials handle local provider access.

gcloud auth application-default login
gcloud config set project my-platform-dev

Secret handling

Stored locally

Connections are saved under the configured projects directory in .iac-studio-connections.json. Treat this as a local secret file when static credentials or service account JSON are used.

Redacted by API design

List, create, update, test, run, and WebSocket responses use public connection views. Secret values are not echoed back to the browser or terminal output.

Scoped to one command

Selected connection variables are injected into the runner process for the requested command only. They do not become generated IaC code.

Static credentials are fallback

Static access keys, service principal secrets, and service account JSON are supported for practical demos and restricted environments, but should be rotated and avoided for day-to-day team workflows when profiles or CLI auth are available.

Semantic Plan Gate

Terraform and OpenTofu plans are saved as tfplan, exported to tfplan.json, and classified before apply. The goal is to turn a long raw plan into a reviewer summary that explains what needs attention before real infrastructure changes run.

Safe

Metadata-only or no-op changes, such as tag, label, annotation, or description updates.

Risky

IAM, network exposure, or stateful resource changes that can alter access, reachability, durability, or spend.

Destructive

Deletes and replacements. Stateful replacements call out backup, snapshot, retention, and restore review.

Unknown

Unsupported actions, missing plan JSON, or changes that do not match a known safe pattern.

Apply behavior

  1. Run Plan. IaC Studio writes the plan artifact and classifier JSON for the active project or environment.
  2. Read the summary. The terminal shows counts and the highest-priority reviewer focus items.
  3. Apply uses the saved plan. Terraform/OpenTofu apply consumes tfplan, so the reviewed artifact is the one being applied.
  4. Acknowledge risk when needed. Risky, destructive, or unknown summaries block apply until the request includes risk_acknowledged:true.

AI Setup

AI is optional. The app still works as a visual editor and runner without an AI provider. When enabled, AI requests are used for chat, topology generation, image topology import, plan fixes, suggestions, and agent workflows.

Local Ollama

ollama pull qwen2.5-coder:7b
./bin/iac-studio --ai-endpoint http://localhost:11434 --ai-model qwen2.5-coder:7b

OpenAI-compatible providers

OPENAI_API_KEY=sk-... \
OPENAI_MODEL=gpt-4o-mini \
./bin/iac-studio

Anthropic

ANTHROPIC_API_KEY=sk-ant-... \
ANTHROPIC_MODEL=claude-opus-4-7 \
./bin/iac-studio
Setting Default Use
--ai-endpoint http://localhost:11434 Ollama or OpenAI-compatible base endpoint.
--ai-model deepseek-coder:6.7b Default model name used by the server.
OPENAI_API_KEY unset Enables OpenAI-compatible API calls.
OPENAI_ENDPOINT https://api.openai.com/v1 Overrides the OpenAI-compatible API endpoint.
OPENAI_MODEL gpt-4o-mini Overrides the OpenAI model.
ANTHROPIC_API_KEY unset Enables Anthropic-backed requests.
ANTHROPIC_MODEL claude-opus-4-7 Overrides the Anthropic model.
Privacy model: local Ollama requests stay on your machine. API providers receive the prompt content needed for the requested action. IaC Studio does not add telemetry or account tracking.

Policy, Security, and Runtime Safety

Local binding

The default server binds to 127.0.0.1. Docker uses 0.0.0.0 inside the container so the host port can map correctly.

Origin checks

CORS and WebSocket origin validation are intended to accept localhost development and app origins only.

Path sandboxing

Project names and file paths are normalized and validated to prevent traversal outside the configured projects directory.

Plan-before-apply

Apply is gated by a successful plan, semantic risk review, and policy findings unless explicitly acknowledged.

Command control

Infrastructure commands run through a safe runner with timeouts, captured logs, and a kill path.

Cloud target context

Selected Cloud Connections are validated before command execution and inject provider env only for that run.

Scanner output

Findings include severity, policy IDs, affected resources, compliance mapping, and remediation text where available.

Troubleshooting

The app screen is blank or black

Open the browser console first. Most blank screens are frontend runtime errors. Refresh after pulling the latest branch, then check the console stack trace for a component and line number. If the error mentions null or undefined data, try reloading the project from disk or opening a fresh project.

Port 3000 is already in use 
./bin/iac-studio --port 3002
Ollama is unavailable

Start Ollama and verify the model exists.

ollama list
ollama pull qwen2.5-coder:7b
Terraform, OpenTofu, or Ansible command fails

Confirm the CLI is installed and available on PATH. Docker images include Terraform and OpenTofu, but local binaries depend on your machine.

A selected cloud connection is not ready

Open the Cloud tab and run the connection test. Missing required fields are reported by name. For local profile, SSO, Azure CLI, and gcloud flows, also verify the matching cloud CLI is logged in and can run a simple account command outside IaC Studio.

Apply is blocked by policy

Review the Policy Studio findings. Fix blocking issues when possible. Only acknowledge a block when you deliberately accept the risk for the current operation.

Docker cannot write project files

Check the mounted host directory permissions. The default container command writes projects under /projects, which should be backed by a writable host directory.

FAQ

Does IaC Studio require a cloud account?

No. The app runs locally. Cloud credentials are only needed when you run real IaC commands that require provider access.

Where are cloud connection credentials stored?

Named connections are stored locally in .iac-studio-connections.json under your configured projects directory. API responses redact secret values.

Does it send telemetry?

No telemetry is built into the app. External AI providers receive only the requests you choose to send to them.

Can I use it without AI?

Yes. The visual editor, parser, generator, command runner, policy, scan, module, and export paths can be used without AI.

Where are projects stored?

By default, projects live in ~/iac-projects. Override this with --projects-dir.

Can I commit generated files?

Yes. IaC Studio writes normal IaC files. Commit them the same way you would commit hand-written Terraform, OpenTofu, Ansible, or Pulumi projects.
